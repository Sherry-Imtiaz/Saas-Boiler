SaaS Boilerplate - Version Context
Version: v0.11.0
Build Name: Audit Logging Expansion and Security Events

Core architecture rule:
Organisation -> Users. Users belong to exactly one organisation and cannot exist without organisation_id. Every tenant-owned document must be scoped to organisation_id.

Development documentation rule:
Every API must be represented in Swagger/OpenAPI and show authentication requirements. Internal/External remains endpoint classification metadata, while v0.11.0 introduces Audit and Security as domain Swagger tags. Each build zip includes static OpenAPI JSON/YAML plus this version context document.

Build position:
This build starts from the uploaded v0.10.2 baseline. The originally discussed compute allocation and compute jobs modules are intentionally skipped for the core boilerplate sequence because they are optional product modules rather than minimum SaaS foundation.

What was added in v0.11.0:

- Added reusable audit/security utility helpers:

New permissions:
- audit.view: View audit logs for the authenticated organisation.
- audit.platform.view: View platform-level and cross-organisation audit logs.
- security.events.view: View security events for the authenticated organisation.
- security.events.platform.view: View platform-level and cross-organisation security events.

Authentication and scope rules:
- Organisation audit log APIs require UserSessionAuth or PersonalAccessTokenAuth and audit.view.
- Organisation security event APIs require UserSessionAuth or PersonalAccessTokenAuth and security.events.view.
- Platform audit log APIs require UserSessionAuth or PersonalAccessTokenAuth and audit.platform.view.
- Platform security event APIs require UserSessionAuth or PersonalAccessTokenAuth and security.events.platform.view.
- Personal access token scope checks continue to apply to internal APIs.

Known limitations / next build notes:
- The audit/security APIs are read-only visibility APIs in v0.11.0.
- Failed login events are only stored when the submitted email maps to an existing user and organisation. This avoids needing a platform-level unaffiliated event store in the current organisation-first model.
- Compute allocation and compute jobs remain skipped from the core foundation and can be added later as optional modules.
- A future v0.12.0 build should expand test coverage and optionally add UI pages for audit/security visibility.

- A user cannot exist without an organisation_id.

- MongoDB is the main software database.

ACTIVE TOKEN MODEL IN v0.6.1
----------------------------
Active token type:
- user_session_token

Active authentication use:
- User logs in through POST /api/auth/login.
- API returns a short-lived bearer token.
- Frontend, admin UI and Postman use this token.
- Internal APIs use user_session_token + RBAC permissions.

Current Postman rule:
- Use the access_token returned by POST /api/auth/login.
- Personal Access Tokens are not implemented yet.

PLANNED TOKEN MODEL
-------------------
- v0.6.2: Personal Access Token Module.
  - User-owned long-lived tokens for Postman, scripts and developer automation.
  - Tokens must be hashed before storage, scoped, expirable, revocable and auditable.

- v0.6.3: Organisation API Token Foundation.
  - Organisation-owned tokens for external/system-to-system integrations.
  - Tokens must be scoped to organisation, features and external API permissions.

- Future: Service Account Tokens.

  - Must be scoped, auditable and revocable.

TOKENRECORDS COLLECTION PURPOSE
-------------------------------
The tokenrecords collection is introduced in this build as the future storage model for long-lived tokens.

Current user_session_token values are stateless and not persisted in tokenrecords.
Future Personal Access Tokens, Organisation API Tokens and Service Account Tokens should use tokenrecords with hashed token values.

TokenRecord important fields:
- organisation_id
- user_id
- created_by_user_id
- token_type
- token_name
- token_hash
- token_prefix
- audience
- scopes
- status
- expires_at
- last_used_at
- revoked_at
- revoked_by_user_id
- metadata

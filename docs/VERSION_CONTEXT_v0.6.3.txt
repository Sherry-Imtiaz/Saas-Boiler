SaaS BOILERPLATE VERSION CONTEXT
Version: v0.6.3
Build Name: Organisation API Token Foundation
Date: 2026-05-21

CORE PRODUCT RULE
Organisation -> Users.
A user belongs to one organisation only.
A user cannot exist without an organisation.
Every tenant-owned document must include organisation_id unless it is explicitly global platform metadata.

DATABASE
MongoDB is the main software database.
Mongoose models currently include:
- OrganisationModel
- UserModel
- RoleModel
- PermissionModel
- AuditLogModel
- TokenRecordModel

ACTIVE AUTHENTICATION MODEL
v0.5.0 introduced native login and signed bearer tokens.
v0.6.1 formalised token type, audience and scope metadata.
v0.6.2 activated Personal Access Tokens.
v0.6.3 activates Organisation API Tokens.

ACTIVE TOKEN TYPES
1. user_session_token
   - Audience: internal
   - Used by frontend, admin UI and Postman during normal user testing
   - Represents a logged-in user

2. personal_access_token
   - Audience: internal
   - Used by a real user for Postman, scripts and user automation
   - Represents a real user
   - Stored as hashed token record
   - Revocable, expirable and auditable

3. organisation_api_token
   - Audience: external
   - Used by external/system-to-system integrations
   - Represents the organisation/integration, not a normal user
   - Stored as hashed token record
   - Revocable, expirable and auditable
   - Tracks last_used_at

4. service_account_token
   - Planned future token type

TOKEN STORAGE RULE
The raw organisation API token is returned once only.
The database stores only token_hash and token_prefix.
A leaked token can be revoked by status update or delete/revoke endpoint.
